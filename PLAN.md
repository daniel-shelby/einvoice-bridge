# einvoice-bridge — Build Plan (v2)

This is the working plan for the project. It captures decisions, the build
order, and the things explicitly out of scope for v1. The README is the
public-facing description; this file is for the people building it.

## 1. Vision

A local-first middleware between a Point-of-Sale (POS) and the Malaysian
LHDN MyInvois platform. The POS hands an invoice to the bridge, gets a
`202 Accepted` immediately, and a background worker handles UBL document
construction, RSA-SHA256 signing, submission, and status reconciliation —
with retries and the LHDN-mandated cancellation window tracked per row.

If LHDN is unreachable, the POS is unaffected: the row stays `Pending` and
is retried with backoff. Nothing is lost.

## 2. Technical Stack

- **Runtime:** Tokio
- **HTTP server:** Axum 0.8
- **HTTP client:** `reqwest` (rustls feature) for LHDN calls
- **Database:** SQLite via SQLx 0.8 (`runtime-tokio-rustls`, `sqlite`,
  `macros`, `time`, `uuid`)
- **Crypto:** `rsa` + `sha2` for signing; `p12` for loading the `.p12`;
  `base64` for encoding
- **XML / JSON:** `quick-xml` for UBL, `serde_json` for the POS payload
- **Time / IDs:** `time`, `uuid` (v7 — time-ordered, friendlier for SQLite)
- **Config:** `dotenvy` + plain env
- **Logging:** `tracing` + `tracing-subscriber`
- **Architecture:** Hexagonal — `domain/` is pure (no IO); `adapters/`
  wraps SQLite, the LHDN HTTP API, and the inbound Axum API.

`rustls` is used for every TLS connection so the project builds cleanly
on Apple Silicon without fighting Homebrew OpenSSL linkage.

## 4. SQLite schema

See `migrations/0001_init.sql` for the canonical version. Three tables:

- `invoices` — one row per POS invoice; tracks lifecycle from `Pending`
  through `Submitted`, `Valid`/`Invalid`, optional `Cancelled`, and
  `Failed` (terminal after retries exhausted).
- `oauth_tokens` — cached LHDN access token per environment.
- `outbox_events` — work queue for the background workers (`submit`,
  `poll`, `cancel`).

Times are stored as `INTEGER` unix seconds for portability.

## 5. Document construction & signing (LHDN-correct, two-stage hash)

LHDN signing is **not** "minify JSON, hash, sign." For UBL JSON/XML:

1. Build the UBL document **without** the `UBLExtensions` and `Signature`
   blocks.
2. Canonicalise (JSON: RFC 8785 / LHDN's variant; XML: C14N exclusive).
3. SHA-256 → `DocDigest`.
4. Sign `DocDigest` with the private key from the `.p12` → `Sig` (base64).
5. SHA-256 of the leaf certificate (DER) → `CertDigest`.
6. Build the `SignedProperties` block, hash it → `PropsDigest`.
7. Re-embed `Sig`, `CertDigest`, `PropsDigest`, signing time, and the
   certificate into `UBLExtensions` + `Signature`.
8. The *final* document is what gets submitted; the `documentHash` in
   the submission envelope is SHA-256 of the final document, base64-encoded.

Module shape (`crates/domain/src/`):

```
canonicalize.rs   // RFC 8785 JSON canonicalisation
digest.rs         // SHA-256 helpers
signer.rs         // P12 loading + RSA-SHA256 sign
ubl.rs            // UBL doc builder + signed-properties embedding
```

All pure functions. Tested with fixtures.

## 6. LHDN HTTP client (`crates/adapters/src/lhdn.rs`)

- Base URL chosen by `LHDN_ENV`:
  - `preprod` → `https://preprod-api.myinvois.hasil.gov.my`
  - `prod` → `https://api.myinvois.hasil.gov.my`
- OAuth2 client_credentials → cache token in `oauth_tokens` table,
  refresh when `expires_at - 60s < now`.
- Endpoints to implement:
  - `POST /api/v1.0/documentsubmissions`
  - `GET  /api/v1.0/documents/{uuid}/details`
  - `PUT  /api/v1.0/documents/state/{uuid}/state` (cancel)
  - `GET  /api/v1.0/taxpayer/validate/{tin}`
- Map LHDN error envelopes → typed `LhdnError` (rejected vs transient
  vs auth).

## 7. API surface (Axum) — v1

| Method | Path                          | Purpose |
|--------|-------------------------------|---------|
| POST   | `/v1/invoices`                | Enqueue an invoice. Returns `202` with internal id and `Pending`. |
| GET    | `/v1/invoices/:ref`           | Current state, plus `qr_url` once `Valid`. |
| POST   | `/v1/invoices/:ref/cancel`    | Request cancellation (only valid within the LHDN window). |
| GET    | `/healthz`                    | DB + LHDN OAuth token sanity. |

The submit handler is **synchronous to the DB only**: insert row in
`Pending`, push an `outbox_events` row, return 202. No LHDN call on the
request path. This is what makes it offline-first.

## 8. Background workers

Three Tokio tasks, each `loop { sleep; tick().await }`:

- **submitter** — pulls due `outbox_events` of kind `submit`, builds and
  signs the UBL, calls LHDN, updates the row to `Submitted` (or `Failed`
  with retry backoff: 30s, 2m, 10m, 1h, capped, max 8 attempts).
- **poller** — for `Submitted` rows, GETs document details until
  `Valid`/`Invalid`; sets `qr_url`, `validated_at`, `cancellable_until =
  validated_at + 72h`.
- **canceller** — handles cancel events; rejects if past
  `cancellable_until`.

Graceful shutdown via `tokio::signal` + a `CancellationToken`.

## 9. Config & secrets

See `.env.example`. The app fails fast at boot if any required variable
is missing or the `.p12` cannot be decrypted.

## 10. Tests

- **Unit:** canonicalisation fixtures (LHDN sample docs in/out), signer
  round-trip with a throwaway key, retry-backoff math.
- **Integration:** spin SQLite in a tempdir, hit Axum with
  `tower::ServiceExt::oneshot`, mock LHDN with `wiremock`.
- **Manual:** preprod sandbox checklist (submit, poll, cancel,
  expired-cancel, invalid-TIN).

## 11. Build order

1. Cargo workspace + migrations + `.env.example` ✅
2. Domain canonicalisation + signer with fixture tests (no DB, no HTTP) ✅
3. SQLx repo + `POST /v1/invoices` returning 202 ✅
4. LHDN client with OAuth token cache (against preprod) ✅
5. Submitter worker end-to-end on preprod sandbox
   - **5a.** Pipeline scaffold: minimal UBL builder, signing flow,
     submitter loop, retry/backoff, `LHDN_OFFLINE` dev escape hatch,
     wiremock-based integration tests ✅
   - **5b.** UBL fidelity to LHDN-acceptance level (parties, lines,
     totals, tax subtotals, full XAdES properties); iterate against
     real preprod sandbox responses
6. Poller + cancel + QR URL ✅
   - Poller picks up `poll` outbox events (enqueued by the submitter on
     success), GETs `/documents/{uuid}/details`, transitions
     `Submitted → Valid/Invalid/Cancelled`, fills `qr_url` and
     `cancellable_until = validated_at + 72h`.
   - `POST /v1/invoices/:ref/cancel` validates state + window and enqueues
     a `cancel` outbox event; the canceller worker calls LHDN's PUT cancel.
7. Hardening: retries, graceful shutdown, `/healthz`, observability

## 12. Out of scope for v1 (track for later)

- Multi-tenant certificate stores
- Consolidated, self-billed, and refund-note document types
- Webhook callbacks to POS (poll-only for now)
- Prometheus metrics (just `tracing` for v1)
